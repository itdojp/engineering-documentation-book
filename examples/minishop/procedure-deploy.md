# 手順書: MiniShop デプロイ（例）

## 目的

- MiniShop（Orders API）を production にデプロイし、正常性を確認する

## 入力（前提）

- 対象環境: production
- 必要権限: deploy 権限、監視閲覧権限
- 依存: メンテ枠 30 分、DBマイグレーション無し（今回）
- 連絡: 影響が出る場合は事前に告知（ステータスページ/社内）

## 手順

1. 準備
   - 直近の変更点（PR/Release）を確認し、影響範囲とロールバック手順を復習する
   - 監視ダッシュボード（エラー率/レイテンシ）を開く
2. 実行
   - リリースを実施する（例: `deploy --env=production --version=v1.2.3`）
3. 検証（成功条件）
   - 外形: `GET /healthz` が `200`
   - メトリクス: 5分間エラー率 < 1%、レイテンシ p95 < 200ms
   - ログ: `TimeoutError` の発生が増加していない（request-id を追跡可能）

## 例外/失敗時

- 失敗パターン: デプロイ後にエラー率が急増（例: > 5%）
- 判断基準: 5分継続で悪化する場合はロールバックを実施
- 連絡先/エスカレーション: On-call → チームリード → 関係部署（runbook.md参照）

## ロールバック

- 戻す範囲: API を直前リリースへ戻す（DB変更がない前提）
- 手順: `deploy --env=production --version=<previous>`（例）
- 注意点: キャッシュ/設定の残骸が影響し得るため、必要なら再起動・キャッシュクリアも実施

## 責任分界

- 実行者: On-call
- 承認者: チームリード
- 影響を受ける利用者: エンドユーザー/CS

# 第10章：Docs-as-Code運用（レビュー/版管理/CI/公開）

## この章で学ぶこと

- 文書をPRで回す運用（レビュー観点/責任分界）を理解する
- CI（lint/link等）で品質ゲートを作る
- 公開/権限/変更履歴の扱いを決める

## この章の成果物（または判断基準）

- Docs-as-Code運用ルール（PRレビュー、Owner、変更履歴）
- 品質ゲート（例: `npm test`）の導入方針

## 本文

Docs-as-Code の狙いは、文書の変更を“レビュー可能”にし、更新の責任を明確化すること。

### 最小の運用

- PRで変更
- レビュー観点（付録チェックリスト）
- CIで形式チェック（lint/link）
- 変更履歴（ChangeLog）
- 公開範囲と権限

CIは“本文の正しさ”までは保証できないが、最低限の破綻（リンク切れ等）を防げる。

## 具体例（悪い例→良い例）

### 悪い例

```md
文書を直接mainにpush
レビューもCIも無い
誰が更新するか不明
```

### 良い例

```md
PRで文書を更新し、レビュー観点で確認
CIで lint/link を実行
Owner と更新頻度（棚卸し）を決める
変更履歴を残す
```

## チェックリスト

- [ ] PRで回す運用になっている
- [ ] レビュー観点がある
- [ ] CIで最低限の品質ゲートがある
- [ ] 変更履歴の方針がある
- [ ] 公開範囲と権限が決まっている

## まとめ

- ドキュメントは成果物であり、判断基準と運用で品質が決まる
- テンプレとチェックリストを“運用に組み込む”ことで事故を減らす
